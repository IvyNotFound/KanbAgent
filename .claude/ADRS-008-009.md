## ADR-008 — Multi-Agent Task Assignment

**Date:** 2026-02-26 | **Status:** accepted | **Author:** arch

### Context

`tasks.agent_assigned_id` is a unique INTEGER FK → 1 agent per task. Some tasks require collaboration between agents from different scopes (e.g. dev-front + dev-back). Ticket duplication is an undesirable workaround.

### Decision

#### 1. New `task_agents` table (many-to-many)

```sql
CREATE TABLE task_agents (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  task_id     INTEGER NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  agent_id    INTEGER NOT NULL REFERENCES agents(id),
  role        TEXT CHECK(role IN ('primary', 'support', 'reviewer')),
  assigned_at TEXT DEFAULT (datetime('now')),
  UNIQUE(task_id, agent_id)
);
CREATE INDEX idx_task_agents_task_id  ON task_agents(task_id);
CREATE INDEX idx_task_agents_agent_id ON task_agents(agent_id);
```

**Roles:** `primary` (main owner), `support` (occasional contribution), `reviewer` (audit). NULL = unspecified.

#### 2. Backward Compatibility — `agent_assigned_id` (renamed from `agent_assigne_id` in T1016/v25)

`tasks.agent_assigned_id` remains the reference for existing queries (BoardView, dbstart, search-tasks, build-agent-prompt). It represents the primary agent.

**Consistency rule:** `task:setAssignees` automatically syncs `tasks.agent_assigned_id` with the assignee having `role='primary'`, or the first in the list if none is `primary`, or `NULL` if the list is empty.

#### 3. DB Migration — schema version 3 → 4

In `migrateDb()`, after existing migrations, create `task_agents` if it doesn't exist. Bump `CURRENT_SCHEMA_VERSION` from `'3'` to `'4'`.

#### 4. New IPC Handlers

**`task:getAssignees`**
- Input: `(dbPath: string, taskId: number)`
- Output: `{ success: boolean, assignees: Array<{ agent_id: number, agent_name: string, role: string|null, assigned_at: string }>, error?: string }`
- Implementation: `SELECT ta.agent_id, a.name as agent_name, ta.role, ta.assigned_at FROM task_agents ta JOIN agents a ON a.id = ta.agent_id WHERE ta.task_id = ? ORDER BY ta.assigned_at ASC`

**`task:setAssignees`**
- Input: `(dbPath: string, taskId: number, assignees: Array<{ agentId: number, role?: string | null }>)`
- Output: `{ success: boolean, error?: string }`
- Behavior: atomic transaction — DELETE task_agents + INSERT + UPDATE tasks.agent_assigned_id
- Validation: `role` must be `null | 'primary' | 'support' | 'reviewer'`
- Location: `src/main/ipc-agents.ts` (registerAgentHandlers)

#### 5. Preload additions `src/preload/index.ts`

```ts
getTaskAssignees: (dbPath: string, taskId: number): Promise<{
  success: boolean
  assignees: Array<{ agent_id: number; agent_name: string; role: string | null; assigned_at: string }>
  error?: string
}> => ipcRenderer.invoke('task:getAssignees', dbPath, taskId),

setTaskAssignees: (dbPath: string, taskId: number, assignees: Array<{ agentId: number; role?: string | null }>): Promise<{
  success: boolean
  error?: string
}> => ipcRenderer.invoke('task:setAssignees', dbPath, taskId, assignees),
```

#### 6. Front-end Impact (specification for dev-front-vuejs)

**`TaskDetailModal.vue`**: multi-select agents with dropdown + role selector per agent. Call `setTaskAssignees` on save. Display primary agent first (role='primary' or agent_assigned_id).

**`TaskCard.vue`**: fetch `task_agents` via `getTaskAssignees` or from store. Display ≤ 3 colored avatars (`agentColor`) + "+N" badge if N > 3. Fallback to `agent_assigned_id` if `task_agents` is empty.

**`BoardView.vue`**: no modification — filtering by `agent_assigned_id` remains valid (primary agent).

**Store `tasks.ts`**: the `assignees` field can be loaded on demand (lazy) in `TaskDetailModal`, not necessarily in the global store (avoid boardview fetch overhead).

### Implementation Order

1. `dev-back-electron`: DB migration + IPC handlers (following this ADR)
2. `dev-front-vuejs`: UI TaskDetailModal + TaskCard
3. `test-back-electron` + `test-front-vuejs`: tests
4. `review`: final audit

### Consequences

- `CURRENT_SCHEMA_VERSION`: `'3'` → `'4'` in `src/main/db.ts`
- `tasks.agent_assigned_id` (renamed from `agent_assigne_id` via T1016 migration v25)
- `CLAUDE.md` (DB schema section) updated to include `task_agents`
- No IPC breaking change — two additive new handlers

## ADR-009 — Replacing node-pty/xterm.js with CLI stream-json

**Date:** 2026-02-26 | **Status:** accepted (Option B — v1.x roadmap) | **Author:** arch

### Context

The current node-pty + xterm.js architecture has recurring issues:

| Issue | Tickets |
|---|---|
| xterm.js memory leaks | T559, T564 |
| PTY security vulnerabilities | T532, T546 |
| Complex PTY lifecycle (orphaned listeners) | T563 |
| Unstructured output — rich display impossible | — |

Two replacement options were evaluated via dedicated spikes.

---

### Option A — SDK @anthropic-ai/claude-agent-sdk (T574 — REJECTED v0.x)

**Evaluation:** the SDK is a wrapper around the Anthropic REST API — it calls `api.anthropic.com` directly, **not** the local `claude` CLI.

**Blocking constraint:** `ANTHROPIC_API_KEY` required. The development environment authentication is **OAuth CLI** (Claude Pro/Max — subscription), not an API key. These two modes are incompatible in SDK v0.x.

**Decision:** REJECTED for v0.x. Kept on v2.0 roadmap if Anthropic exposes an OAuth-compatible SDK or an SDK that delegates to the local CLI.

---

### Option B — CLI stream-json via child_process.spawn (T576 — VALIDATED)

**Hypothesis:** `claude -p --verbose --input-format stream-json --output-format stream-json` with `stdio: ['pipe', 'pipe', 'pipe']` = complete replacement without API key, existing OAuth auth.

#### Spike T576 Results

| Question | Test | Result |
|---|---|---|
| Q1 | stream-json without `-p` (pure interactive mode) | **FAIL** — `--output-format` requires `-p` |
| Q1b | stream-json with `-p --verbose` (baseline) | **PASS** ✓ |
| Q2 | Multi-turn via stdin `--input-format stream-json` | **PASS** ✓ |
| Q3 | `--thinking` exposes thinking blocks in stream-json | **FAIL** — no `type:"thinking"` block on claude-sonnet-4-6 |
| Q4 | `--resume <conv_id>` compatible with stream-json | **PASS** ✓ |

**Verdict:** partial PASS — the 3 critical features (structured launch, multi-turn, session resume) are validated. Q3 (thinking) is a documented limitation, not blocking.

#### Observed JSONL Message Types

| Type | Subtype / fields | Moment |
|---|---|---|
| `system` | `subtype: "init"`, `session_id` | Startup — provides the `conv_id` |
| `user` | `message.role: "user"`, `message.content[]` | Echo of message sent via stdin |
| `assistant` | `message.content[]` with `type:"text"` | Claude response (text) |
| `assistant` | `message.content[]` with `type:"tool_use"` | Tool call |
| `result` | `cost_usd`, `num_turns`, `session_id` | End of each turn |

> **Note Q3:** `message.content[]` with `type:"thinking"` absent in claude-sonnet-4-6 responses even with `--thinking enabled`. Only Opus might expose it (untested). Workaround: keep current PTY thinking mode (ADR-002) for agents that need it.

#### stdin Format (multi-turn)

```json
{
  "type": "user",
  "message": {
    "role": "user",
    "content": [{ "type": "text", "text": "..." }]
  }
}
```

> The **first** stdin message must be sent immediately after `spawn()`, before even the `system:init` event. Subsequent messages are sent after each `result` event.

> If no stdin message planned → close stdin with `proc.stdin.end()` immediately after spawn.

> `--verbose` is **required** — without this flag, `system` and `user` events are absent from the JSONL stream.

#### Target Architecture

```
child_process.spawn('claude', [
  '-p', '--verbose',
  '--input-format', 'stream-json',
  '--output-format', 'stream-json',
  // optional: '--resume', convId
  // optional: '--append-system-prompt', '...'
], { stdio: ['pipe', 'pipe', 'pipe'] })
```

**In `src/main/ipc.ts` or `src/main/terminal.ts`:**
- `stdout`: readline line by line → parse JSON → emit via structured IPC
- `stdin`: JSONL write in format `{ type: "user", message: ... }`
- `session_id` extracted from `system:init` event → stored in `sessions.claude_conv_id`

**IPC Electron → Vue (new channels):**

```ts
// agent:stream — emitted for each parsed JSONL event
interface AgentStreamEvent {
  terminalId: string
  event: {
    type: 'system' | 'user' | 'assistant' | 'result'
    subtype?: string
    session_id?: string
    message?: { role: string; content: Array<{ type: string; text?: string }> }
    cost_usd?: number
    num_turns?: number
  }
}

// agent:sendMessage — sends a user message mid-session
interface AgentSendMessagePayload {
  terminalId: string
  text: string
}
```

**Vue — `AgentStreamView.vue` component** (replaces `TerminalView.vue` for agent sessions):
- Displays `assistant.message.content[]` blocks in a structured way (text, tool_use)
- Maintains a scrollable turn history
- "Send" button → IPC `agent:sendMessage`

#### Subscription Constraint

| Auth | Compatibility |
|---|---|
| OAuth CLI (Claude Pro/Max) | ✓ Compatible — `claude -p` uses the same auth as interactive CLI |
| ANTHROPIC_API_KEY | ✓ Compatible (if defined in env) |
| SDK @anthropic-ai/sdk direct | ✗ Incompatible with OAuth (see Option A) |

---

### Decision

**Adopt Option B — CLI stream-json** to replace node-pty/xterm.js in Claude agent sessions.

**Roadmap:**
- **v1.x**: main implementation — `child_process.spawn + stream-json` in `terminal.ts` or new `agent-stream.ts`, structured IPC, `AgentStreamView.vue`
- **v2.0**: re-evaluate Option A SDK if Anthropic publishes an OAuth-compatible SDK

### Open Questions (non-blocking for v1.x)

1. **Q3 — Thinking blocks:** no `type:"thinking"` in stream-json on sonnet-4-6. Keep node-pty for agents with `thinking_mode:'auto'` until resolved, or accept the absence of thinking blocks in the structured UI.
2. **TerminalView coexistence:** generic terminals (non-agent) continue to use node-pty + xterm.js. The migration is **additive** — `AgentStreamView` for agent sessions, `TerminalView` for the rest.
3. **System prompt injection:** with stream-json, `--append-system-prompt` works as a CLI argument. Validate that special characters (backticks, quotes) are correctly escaped.

### Required Modifications

| File | Action | Effort |
|---|---|---|
| `src/main/agent-stream.ts` (new) | Spawn + readline + IPC for CLI stream-json | High |
| `src/main/ipc.ts` | Register `agent:*` handlers | Medium |
| `src/preload/index.ts` | Expose `agentStream`, `agentSendMessage`, `agentKill` | Low |
| `src/renderer/src/components/AgentStreamView.vue` (new) | Structured display of stream-json events | High |
| `src/renderer/src/stores/tabs.ts` | Tab type `'agent-stream'` | Low |
| Tests | `agent-stream.spec.ts` + `AgentStreamView.spec.ts` | Medium |

### Consequences

- node-pty + xterm.js kept for generic terminals — non-breaking migration
- `sessions.claude_conv_id` populated from `system:init.session_id` — no schema change
- `--resume` validated — session resume works with stream-json
- Implementation tickets to be created by `review`: `agent-stream.ts` (back-electron) + `AgentStreamView.vue` (front-vuejs) + tests
